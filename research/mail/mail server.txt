任務：整合 Mail Server / SMTP 郵件系統

目標：
讓 hackme_web 支援：
1. 帳號註冊 Email 驗證
2. 忘記密碼 / 重設密碼驗證
3. root 伺服器緊急狀況通知
4. 安全事件 Email alert

核心原則：
- 不要自行實作 SMTP server，先支援外部 SMTP relay。
- SMTP 密碼不可明文回傳。
- token 必須一次性、短效、可撤銷。
- Email 失敗不得導致伺服器崩潰。
- 所有寄信事件需寫入 audit / security event。
- 不得把 password reset token 寫入 log。

四、新增 service

新增：

services/mail_service.py

功能：

1. load_mail_config(conn)
- 讀取 system_settings
- 驗證 host/port/from/public_base_url
- secret 不進 log

2. mask_email(email)

- 同上
- token TTL 短，預設 15 分鐘

7. send_email(...)
- 使用 smtplib
- 支援 STARTTLS / SSL
- timeout
- exception handling
- 不印出 SMTP password

8. enqueue_email(...)
- 寫入 email_outbox

9. process_email_outbox(...)
- 從 queued 送信
- attempts +1
- sent / failed
- last_error masked

10. send_email_verification(...)
11. send_password_reset_email(...)
12. send_root_alert(...)

六、註冊 Email 驗證流程

若 require_email_verification = true：

註冊流程：
1. 使用者註冊
2. 建立帳號，但狀態為 pending_email_verification 或 email_verified=0
3. 產生 verification token
4. 寄出驗證信
5. 未驗證前：
   - 可登入與否由系統設定決定
   - 預設不允許發文、留言、私訊、上傳
6. 使用者點擊：
   GET /api/auth/verify-email?token=...
7. 驗證成功：
   - email_verified=1
   - email_verified_at=now
   - token used_at=now
   - 寫 audit event

新增 API：
- POST /api/auth/resend-verification-email
- GET /api/auth/verify-email

安全：
- resend 要 rate limit
- verify token 一次性
- token 過期不可用
- 回應避免帳號列舉

七、忘記密碼流程

新增 API：

POST /api/auth/forgot-password
Request：
{
  "email": "..."
}

行為：
- 無論 email 是否存在，回覆一致：
  「若帳號存在，重設信已寄出」
- 若存在且未被封鎖：
  - 建立 password_reset_token
  - 寄出 reset email
- rate limit by IP/email
- 寫 security_event，但不要暴露帳號是否存在

GET /reset-password?token=...
- 前端頁面顯示新密碼表單

POST /api/auth/reset-password
Request：
{
  "token": "...",
  "new_password": "..."
}

行為：
1. hash token 查 DB
2. 檢查未過期、未使用
3. 檢查 password policy
4. 更新 password_hash
5. 將 token used_at=now
6. revoke 該使用者所有 active sessions
7. 設 must_change_password=0
8. 寫 audit / security_event
9. 寄 root alert 可選：若短時間大量 reset request

安全：
- 不可把 token 存 log
- 成功後立即失效
- 重設密碼後登出所有 session
- 對 disabled/suspended 帳號要有一致回應，但不可允許繞過處分

八、root 緊急通知

新增 alert service hook：

在以下情境寄信給 root：

critical：
- Integrity Guard high-risk finding
- audit chain verification failed
- snapshot restore failed
- superweak mode entered
- superweak dirty state kept
- repeated root login failure
- storage root validation failed
- DB integrity check failed
- Discord webhook/token abuse detected
- secrets scan high-risk finding if CI report imported

warning：
- daily snapshot failed
- email outbox repeated failure
- unusual login spike
- pending high-risk upload
- maintenance mode enabled

寄送規則：
- mail_root_alert_enabled = true 才寄
- severity >= mail_root_alert_min_severity
- recipient 使用 mail_root_alert_recipients
- 若未設定，使用 root 帳號 email
- 同 alert_type 需有 cooldown，避免 mail storm
- 例如 30 分鐘內同類 critical 只寄一次，其他寫入 email_alert_events

十、背景處理

MVP 可先不引入 Celery。

方案：
- 每次 enqueue 後可嘗試同步送出一次
- 另提供 root 手動 process outbox
- server startup 可啟動簡單 background worker，每 60 秒處理 queued
- worker 必須可關閉
- 測試環境不得真的寄信，可使用 dummy backend

十三、驗收標準

完成後需證明：

1. root 可在 UI 設定 SMTP。
2. SMTP password 不會被 API 回傳。
3. root 可送 test email。
4. require_email_verification=true 時，新註冊帳號需驗證 email。
5. 驗證連結過期或重複使用會失敗。
6. 忘記密碼不會造成帳號列舉。
7. reset password 成功後舊 session 失效。
8. Integrity Guard high-risk finding 可觸發 root email alert。
9. mail server 壞掉時，主流程不 crash。
10. 所有 token 不進 log，只存 hash。

十四、禁止事項

- 不要自行寫 SMTP server。
- 不要把 raw token 存 DB。
- 不要把 SMTP password / reset token / verification token 寫入 log。
- 不要讓忘記密碼 API 暴露 email 是否存在。
- 不要讓未驗證 email 的帳號取得完整互動權限。
- 不要讓 suspended 帳號用 reset password 繞過處分。
- 不要用永久有效 token。
