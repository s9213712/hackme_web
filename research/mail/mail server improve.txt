任務：升級 Mail 系統為營運級（整合 Phase 2–5：可靠性 + 安全通知 + 使用者偏好 + 多供應商 + 監控）

目標：
- 郵件傳送「可靠可觀測」（queue / retry / dead-letter）
- 完整「安全通知中心」（root/使用者關鍵事件）
- 使用者「通知偏好 + 退訂」與「摘要信」
- 「多 SMTP provider fallback」與退信/投訴處理
- 營運儀表板（rate / failure / queue / alert storm 控制）

核心原則：
- SMTP 憑證與 token 不得明文回傳或寫入 log
- 郵件失敗不得影響主流程（非同步處理）
- 所有 token 一次性、短效、只存 hash
- 重要安全通知不可被使用者關閉
- 全部寄信行為需可審計（audit + events）

────────────────────────────────
四、寄信流程（統一）
────────────────────────────────

所有寄信改為：
enqueue_email → outbox → worker → provider → delivery_logs

禁止直接同步 send_email（除 test endpoint）

Worker：
- 每 60 秒跑 process_outbox_batch
- 只處理 queued 且 next_retry_at <= now
- sending → sent / failed
- failed → retry or dead

五、安全通知（Phase 3）
────────────────────────────────

觸發寄信：

CRITICAL：
- Integrity Guard high risk finding
- DB integrity failed
- snapshot restore failed
- audit chain broken

WARNING：
- repeated login failure
- unusual IP login
- upload high risk

USER：
- login from new IP
- password changed
- email changed
- reset password success

規則：
- notify_security 永遠 true
- 其他依 user_email_preferences
- root alert 走 root 設定 recipients

────────────────────────────────
十二、驗收標準
────────────────────────────────

1. 郵件失敗會 retry，不影響主流程
2. 超過上限進 dead_letter
3. root 可收到 critical alert
4. 使用者可設定通知偏好
5. unsubscribe link 可用
6. digest email 可正常寄出
7. provider fallback 可切換
8. dashboard 可顯示 mail 狀態
9. 不會發生 mail storm
10. 所有敏感資訊不出現在 log/API

────────────────────────────────
十三、禁止事項
────────────────────────────────

- 不要同步 send_email（必須走 queue）
- 不要記錄 SMTP password / token
- 不要讓 user 關閉 security 通知
- 不要無限制 retry
- 不要忽略 bounce/complaint
- 不要讓 mail failure 影響主要功能
