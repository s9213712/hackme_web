# ComfyUI QoS Regression Check

## Scope

Investigated why ComfyUI generation reduced hackme_web service quality even when the image was eventually generated successfully.

## Confirmed Cause

Completed ComfyUI jobs stored generated image previews as inline `data_url` base64 in `comfyui_generation_jobs.result_json`.

Observed live DB rows before cleanup:

- `361948efdb60dab4d6a6c66f`: `result_json` about 3.6 MB.
- `0aa46b6c6fb95d79101f0974`: `result_json` about 2.8 MB.
- `9a96b49094ca344e0fcc271e`: `result_json` about 6.0 MB.

The job status API then returned these multi-MB payloads through the main web workers, causing avoidable SQLite writes, JSON serialization, memory use, and response transfer spikes.

## Fix

- ComfyUI generation and workflow jobs now request output refs without fetching full output bytes into the job result.
- Job result payloads store `image_ref`, MIME type, size metadata, billing, and history information only.
- Frontend generation, rerun, and workflow preset flows lazy-load preview images through `/api/comfyui/image-preview` only when the UI needs to display them.
- Job persistence and job status serialization strip any future inline `data_url` values defensively.
- Existing oversized job rows were pruned from MB-scale JSON to sub-KB JSON.

## Remaining Risk

ComfyUI still competes for machine-level CPU, RAM, disk, and GPU/VRAM while generating. This patch removes the confirmed web-app payload amplification, but large model loads can still reduce overall host quality if the ComfyUI process saturates hardware.

Also observed several residual hackme_web test servers on other ports. Those should be stopped before future performance comparison runs to avoid contaminating measurements.
