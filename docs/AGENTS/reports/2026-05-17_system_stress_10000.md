# 2026-05-17 10000 Logical User System Stress

## Scope

This pass used an isolated `test_for_develop.sh` runtime, not the developer's
real runtime DB.

Targeted surfaces:

- DB read/write paths, session/auth, notifications, Job Center.
- Cloud drive list/upload/download and resumable upload start.
- Video list/playback, share management, and HLS manifest access.
- HuggingFace/diffusers-facing image generation routes exposed through the
  current `/api/comfyui/*` API names.
- BT/direct-link reject paths.
- Trading markets, dashboard, and grid preview.
- Games catalog and chess leaderboard.
- Community, chat/PM room list, and violation-like bad requests.
- Continuous `/api/version` QoS probe during load.

The formal run modeled 10000 logical users/operations with 512 active request
workers and 512 cloned authenticated sessions. It intentionally did not keep
opening independent login sessions during the functional run, because the first
login-storm pass mainly measured rate-limit pressure instead of feature
bottlenecks.

## Result

Status: FAIL / degraded.

The server did not immediately crash, but service quality collapsed before a
true 10000-active-socket profile was necessary. At 512 active concurrent
requests, `/api/version` repeatedly timed out, transport failures dominated the
run, and the Flask process expanded to more than 3000 request threads during the
peak.

## Findings

1. High: request concurrency, not DB locking, is the first confirmed bottleneck.

   The process temporarily grew to roughly `3167` threads under pressure. The
   built-in Flask/Werkzeug server path is not suitable for this profile because
   active requests can pile up into memory and thread pressure. This explains
   why health checks timed out even though WAL sizes stayed modest and the DB
   did not report lock failures.

2. High: service quality degrades before a 10000-active-client test is useful.

   A 512-concurrency run already produced an `87.6%` transport/5xx-style failure
   rate and `/api/version` p95 above 8 seconds. Raising active concurrency to
   10000 on this same process would mostly test OS/thread exhaustion and could
   obscure the already confirmed application bottleneck.

3. High: memory is not released after the stress wave.

   The server recovered enough to answer `/api/version` after the run, but RSS
   stayed around `2.8 GB`. This is dangerous for a small VPS because repeated
   traffic spikes can ratchet memory upward even when the site appears alive
   again.

4. Medium: functional endpoint p95 is seconds even in the small clone smoke.

   The 240-op clone smoke had no transport failures and QoS stayed healthy, but
   overall p95 was still `3763 ms`. Slow examples included chess leaderboard,
   community boards, trading dashboard, chat rooms, notifications, and drive
   list. These endpoints need endpoint-level profiling after the serving model
   is hardened.

5. Medium: HuggingFace image generation coverage was intentionally capped.

   The stress tool hit status/quote and capped generation attempts at 20. This
   avoids accidentally starting many expensive image-generation jobs. A separate
   worker-level HuggingFace queue/load test is still needed once the production
   provider, token, model cache, and concurrency limits are finalized.

6. Medium: video/HLS/share access was exercised, but not proven as a clean 200
   playback path under the 10000 run.

   Under load most video/HLS operations timed out. Browser playback
   compatibility was already covered separately by the browser video report; the
   next stress pass should use a pre-warmed published video with known-ready HLS
   assets and measure direct segment throughput.

7. Expected: login storm is separately protected by rate limiting.

   The first 10000 run created many independent login attempts and quickly hit
   `429` responses. That is useful as a rate-limit confirmation, but not a fair
   feature stress profile. The clone-mode run is the relevant functional result.

## Recommendations

1. Put the app behind a bounded production WSGI/ASGI server for stress testing.

   Use a fixed worker/thread model such as gunicorn or waitress, with explicit
   worker count, thread count, request timeout, keep-alive timeout, and max
   request recycling. The development server path should not be the benchmark
   target for 10000-user claims.

2. Add global request backpressure.

   When active requests exceed the safe pool size, return a quick `503`/retry
   response for non-critical routes instead of letting thousands of threads and
   sockets accumulate.

3. Split critical health and root/status routes from heavy feature routes.

   `/api/version`, root status, background job status, and trading safety status
   should remain responsive even while upload/download/video/community/game
   routes are saturated.

4. Add endpoint-level profiling after the serving model is bounded.

   Prioritize:
   - `community_boards`
   - `chess_leaderboard`
   - `trading_dashboard`
   - `chat_rooms`
   - `drive_list`
   - `notifications`

5. Keep DB improvements, but do not treat SQLite as the current first failure.

   WAL and DB sizes stayed controlled, and no lock signature appeared in logs.
   The DB work remains valuable, but this 10000-system pass shows request
   concurrency/memory/backpressure is currently the earlier bottleneck.

6. Run the next 10000-user test in two stages.

   First stage: bounded production server profile, for example 4 workers x 16
   threads, with strict timeouts and queue limits. Second stage: endpoint-specific
   mixes for upload/download/HLS/trading/chat/games after health checks stay
   below the QoS budget.
