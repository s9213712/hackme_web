# 2026-05-16 1000 Logical Online DB Stress

Scope: isolated runtime DB stress for a 1000 logical online user profile. This
did not use the developer's real runtime DB.

## Code changes covered by this pass

- SQLite helpers now expose read-only connections for main/auth/audit/control DBs.
- Runtime SQLite PRAGMAs now include configurable `cache_size`, `mmap_size`,
  `journal_size_limit`, and `wal_autocheckpoint`.
- Auth token validation now reads sessions through a read-only auth connection
  and opens a write connection only when revoking a session or refreshing
  `last_seen`.
- Session `last_seen` refresh is throttled with per-session interval + jitter:
  `HTML_LEARNING_SESSION_LAST_SEEN_REFRESH_INTERVAL` and
  `HTML_LEARNING_SESSION_LAST_SEEN_REFRESH_JITTER_SECONDS`.
- Account-session GET/admin user-list session reads now prefer the read-only
  auth DB helper.
- The DB stress probe now records CPU, memory, RSS, disk, WAL size, page count,
  freelist count, and per-DB journal mode while the test is running.
- The DB stress probe can model 1000 logical online users and can tune or
  disable synthetic session touches with `--online-session-touch-every`.

## Profile A: direct heartbeat worst case

Command shape:

```bash
PYTHONPATH=/home/s92137/hackme_web python3 scripts/testing/db_stress_probe.py \
  --duration 10 --threads 1100 --user-count 1000 --online-users 1000 \
  --online-write-every 4 --main-writers 4 --user-writers 12 \
  --user-readers 32 --job-writers 8 --auth-writers 4 --session-writers 12 \
  --audit-writers 4 --control-writers 2 --readers 8 --external-workers 4 \
  --backend file --flush-interval 1.5 --event-flush-interval 5.0 \
  --monitor-interval 1.0
```

Result:

- `ok=true`
- `error_count=0`
- `lock_error_count=0`
- logical online users: `1000`
- external job-progress workers: `4/4` succeeded

Resource summary:

- CPU avg/max: `44.01% / 53.80%`
- monitored RSS max: `330.32 MB`
- min available memory: `9227.66 MB`
- peak WAL: main `1.890 MB`, auth `0.813 MB`, audit `0.460 MB`,
  control `0.393 MB`

Key p95 latency:

- external worker `job_progress`: about `1.1 ms`
- `mixed_read`: `89.060 ms`
- `online_profile_read`: `118.682 ms`
- `user_profile_read`: `102.668 ms`
- `online_activity_write`: `3161.573 ms`
- `online_session_touch`: `8592.475 ms`
- `session_write`: `4166.860 ms`
- `auth_write`: `4145.644 ms`

Finding: lock-induced 500 risk did not reproduce, but direct heartbeat writes
from 1000 online users create a long auth/session writer queue. This is why the
runtime path must not refresh `last_seen` on every request.

## Profile B: coalesced session model

Command shape:

```bash
PYTHONPATH=/home/s92137/hackme_web python3 scripts/testing/db_stress_probe.py \
  --duration 10 --threads 1100 --user-count 1000 --online-users 1000 \
  --online-write-every 8 --online-session-touch-every 0 \
  --main-writers 4 --user-writers 12 --user-readers 32 --job-writers 8 \
  --auth-writers 2 --session-writers 2 --audit-writers 4 \
  --control-writers 2 --readers 8 --external-workers 4 \
  --backend file --flush-interval 1.5 --event-flush-interval 5.0 \
  --monitor-interval 1.0
```

Result:

- `ok=true`
- `error_count=0`
- `lock_error_count=0`
- logical online users: `1000`
- external job-progress workers: `4/4` succeeded

Resource summary:

- CPU avg/max: `49.13% / 60.15%`
- monitored RSS max: `292.54 MB`
- min available memory: `9105.73 MB`
- peak WAL: main `1.898 MB`, auth `0.314 MB`, audit `0.436 MB`,
  control `0.314 MB`

Key p95 latency:

- external worker `job_progress`: about `1.1 ms`
- `mixed_read`: `61.059 ms`
- `online_profile_read`: `132.028 ms`
- `user_profile_read`: `113.329 ms`
- `auth_write`: `180.521 ms`
- `session_write`: `191.984 ms`
- `audit_write`: `223.624 ms`
- `control_write`: `157.097 ms`
- `online_activity_write`: `9241.335 ms`

Finding: after synthetic per-request session writes are removed, auth/session
latency becomes acceptable for this SQLite deployment. The remaining severe
queue is main DB activity/counter writes, which is the next high-frequency DB
hotspot.

## Interpretation

1. The current SQLite hardening is doing its primary job: the 1000-user profiles
   produced no `database is locked` errors and no lock-like 500 condition in the
   isolated stress harness.
2. The file progress backend is effective for high-frequency latest-progress
   state. Four external workers produced more than 50k progress updates in
   10 seconds with p95 around 1 ms, while DB event rows stayed small.
3. Auth/session must stay coalesced. A direct heartbeat write per user/request
   is not viable on a small SQLite server.
4. Main DB user activity / counters are now the hottest write path in the
   stress model. If this pattern appears in production features, it should be
   buffered, snapshotted, or moved to a dedicated worker/checkpoint path.
5. Redis can help later as a volatile progress/presence/cache backend, but it
   should not become the source of truth for final job state, financial data,
   trading settlement, or PointsChain ledger records.

## Next hardening candidates

- Add a generic buffered counter/presence service for non-financial counters:
  view counts, presence, lightweight activity ticks, and latest-progress style
  UI state.
- Keep final user actions, moderation, financial entries, trading settlement,
  and audit events synchronous and durable.
- Consider a dedicated presence backend interface using the same pattern as
  `ProgressBackend`: file fallback now, Redis optional later.
- If sustained 1000 real concurrent writers is a production requirement, move
  the write-heavy truth tables to PostgreSQL or put writes behind a durable
  worker queue. SQLite can be made resilient, but it remains single-writer.
