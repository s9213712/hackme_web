# Rules For Agents

這份文件定義 `hackme_web` 的全專案工作原則。未來只要新增、修改或重構任何
功能，都不能只完成程式本體；文件、測試、錯誤回饋、手機版體驗、伺服器端
驗證與交付回報必須同步完成。

這不是建議，而是完成定義的一部分。若以下任一項未完成，不得宣稱功能已完整
完成。

## 適用範圍

- 適用所有後端、前端、資料庫、腳本、文件、測試、部署與 QA 變更。
- 適用新功能、修 bug、重構、設定調整、權限規則變更與 UI 調整。
- 適用 root / 管理員工具、一般使用者功能、營運腳本與安全控制。

## 1. 文件同步更新

每次功能變更都必須同步檢查並更新相關文件。

- 更新所有相關 README。
- 更新使用教學、管理員教學、開發者文件。
- 更新 API 文件、資料表說明、設定說明。
- 若功能有前端入口，必須補充操作流程與範例。
- 若功能有安全限制，必須說明權限、風險、錯誤處理方式。

最低要求：

- 一般使用者要知道怎麼使用。
- 管理員 / root 要知道怎麼設定、監看與排錯。
- 開發者要知道 API、資料流、驗證邏輯與測試入口。

## 2. 測試同步更新

每次功能變更都必須同步檢查並更新測試。

- 新增或修改對應測試腳本。
- 更新 smoke test、pre-push check、QA runbook。
- 重要功能必須有正常流程、錯誤流程、權限不足流程、邊界值流程測試。
- 測試不得污染正式 runtime、DB、log、cache。
- 測試結果必須可重跑、可追蹤、可產生證據。

最低要求：

- 關鍵路徑至少要能用自動測試重跑。
- 涉及權限、金流、積分、交易、安全、刪除、snapshot/restore 的功能，
  必須有更高強度的回歸測試。
- 若沒有補測試，必須明確說明原因與風險，不能靜默略過。

## 3. 禁止靜默失敗

所有失敗都必須有清楚回饋，但不能洩漏敏感資訊。

- 所有失敗都必須有明確提醒。
- 後端錯誤要回傳清楚但不洩漏敏感資訊的錯誤訊息。
- 前端不得只顯示「失敗」或完全沒反應。
- 使用者必須知道：
  - 發生什麼事
  - 為什麼可能失敗
  - 下一步可以怎麼做
- 管理員或 root 必須能在 log / audit event 中查到原因。

最低要求：

- 前端要有成功、失敗、處理中、權限不足、資料不存在等狀態提示。
- 後端要有對應 audit / log 線索供管理端查核。
- 錯誤字串不得只是工程內部例外堆疊原文直接外露給使用者。

## 4. 使用者體驗要求

任何新增 UI 或流程都必須考慮一般使用者，而不是只讓工程師看得懂。

- 介面必須人性化，不可只做工程師看得懂的 UI。
- 文字要使用一般使用者能理解的說法，避免過度技術術語。
- 必須提供必要的附加解說、提示、範例、警告。
- 重要操作要有確認提示。
- 危險操作要有二次確認與後果說明。
- 成功、失敗、處理中、權限不足、資料不存在等狀態都要有清楚 UI 回饋。

最低要求：

- 使用者應能從畫面理解功能用途、限制與下一步。
- 危險操作要能清楚看懂後果，而不是只看到按鈕名稱。
- root / 管理頁也不應只堆砌工程用字與裸 API 欄位。

## 5. 手機版優化

所有新增頁面與元件都必須支援手機版，不得只在桌面版可用。

- 所有新增頁面與元件必須支援手機版。
- 不得只在桌面版正常。
- 必須檢查：
  - 小螢幕排版
  - 按鈕大小
  - 表格是否可橫向捲動或改成卡片
  - 側邊選單收合
  - 表單輸入體驗
  - 錯誤訊息不溢出
  - 操作區不被遮擋

最低要求：

- 新增 UI 要有手機尺寸檢查結果。
- 若某些管理表格無法完全手機友善，必須至少提供可捲動與不破版的退化方案。

## 6. 運算責任原則

可信運算必須在伺服器端完成，前端只負責顯示、互動與輔助提示。

- 能在伺服器端完成的運算，不要丟給客戶端計算。
- 客戶端只負責顯示、互動、輸入驗證與必要的即時提示。
- 涉及金額、積分、權限、交易、排名、結算、安全判斷、審計、風控的邏輯，
  必須由伺服器端計算與驗證。
- 前端顯示的結果不得被視為可信資料。
- 後端必須重新驗證所有前端送來的資料。

最低要求：

- 前端送來的數字、價格、權限旗標、流程狀態都不能直接信任。
- 後端必須重新驗證條件、重新計算結果、重新檢查權限。

## 7. 完成交付標準

每次功能完成後，回報必須明列以下項目：

- 修改了哪些功能
- 更新了哪些 README / 文件
- 更新了哪些測試
- 新增了哪些錯誤提醒
- 手機版檢查結果
- 哪些計算已移到伺服器端
- 是否有尚未完成或需要後續處理的項目

若上述任一項缺漏，回報中必須清楚標示為未完成項，不得用「已完成」模糊帶過。

## 實作檢查清單

每次送出功能前，至少自問以下問題：

1. 有沒有同步更新使用者文件、管理文件、開發文件？
2. 有沒有同步補上正常、錯誤、權限不足、邊界值測試？
3. 前端失敗時，使用者是否知道原因與下一步？
4. root / 管理端是否能從 audit 或 log 查到線索？
5. 手機版是否仍可操作？
6. 關鍵計算是否仍留在伺服器端？
7. 最終回報是否交代完功能、文件、測試、風險與未完成項？
