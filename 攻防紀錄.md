## 二、紅隊攻擊紀錄

> 來源：`attack_test/red_team_report.md`｜測試者：紅隊｜目標：`http://localhost:5000`

### 2.1 攻擊任務
1. 驗證是否可透過登入 API 越權登入未授權帳號
2. 壓制測試速率限制、封鎖機制、CSRF 與 session 行為
3. 邊界/異常輸入測試（type confusion、payload 異常、CORS、路徑探測）
4. 敏感檔案與錯誤訊息資源滲透

### 2.2 攻擊成效

| 攻擊項目 | 結果 | 說明 |
|----------|------|------|
| 直接登入未授權帳號 | ❌ 未成功 | 所有未授權帳號回傳統一錯誤訊息 |
| 取得有效 session/token | ❌ 未成功 | 401/403 正常，無 token 洩漏 |
| SQLi 繞過登入 | ❌ 未成功 | `"' OR 1=1 --"` 等 payload 無效 |
| 路徑遍歷/敏感檔案 | ❌ 未成功 | `/api/debug`, `/.git/config`, `../` 均 404 |
| 隱藏未授權路由枚舉 | ❌ 未發現 | 管理/除錯端點均 404 或 401/403 |
| CSRF 繞過 | ❌ 未成功 | CSRF double-submit 机制完整 |

### 2.3 攻擊成效摘要

- 未能成功「**直接登入未授權帳號**」
- 未能在白名單帳號以外取得有效 session / token
- 未能透過 SQLi、路徑遍歷、隱藏端點拿到未授權資料
- 依現階段測試結果：**未見完整權限突破**

### 2.4 紅隊發現（高風險）

#### 🔴 高風險 1：`POST /api/login` 未處理非字串 `username`，回傳 500

- **現象**：`{"username":123,"password": "***"}`、`{"username":null,...}`、`{"username":["x"]}`、`[1,2,3]` 均回 `500`
- **根因分析**：`login()` 在 `request.get_json(force=True)` 後，直接對非 dict 走 `except` 路線回 400，但當 `username` 為 non-string 時，`isinstance(data, dict)` 通過，後續 `data.get("username","")` 取值後 `.strip()` 會在非字串上觸發 AttributeError
- **影響**：
  - 不進入 `fail_log` 失敗計數，不觸發 IP 封鎖
  - 可大量噴送 malformed 請求耗盡資源或作為 DoS 向量
  - 進入例外錯誤頁，未經正常錯誤路徑處理
- **可復現性**：高，穩定觸發

#### 🔴 高風險 2：`X-Forwarded-For` 來源分片，稀釋 IP 速率限制

- **現象**：更換不同 `X-Forwarded-For` header 可繞過同一 IP 的節流計數
- **根因分析**：`get_client_ip()` 在 `remote_addr` 為信任代理（`127.0.0.1`）時信任 `X-Forwarded-For` 第一段；本地測試中攻擊者可不斷置換 XFF 值稀釋計數
- **影響**：
  - `register` 10次/分鐘節流可被多重 XFF 快速稀釋
  - 登入失敗封鎖（3次後封鎖10分鐘）可被 XFF 分片繞過
  - 本地環境中「速率防護」形成高並行測試窗口

#### 🔴 高風險 3：malformed 請求與鎖定機制脫鉤

- **現象**：大量 malformed payload (`[1,2,3]`) 噴向 `/api/login` 後，同來源在同 XFF 下測試有效帳密仍可成功登入（未被 malformed 計數觸發鎖定）
- **根因**：malformed 輸入繞過 `record_failed_login()` 邏輯，導致 fail_log 無累積
- **影響**：攻擊者可用「先噴錯（不入計數）再試正確 payload」循環繞過速率限制

### 2.5 紅隊測試結果（未成功項目）

- `register` 在正常 JSON + 合法條件下可成功，且受節流保護；
- `'/api/login'` 的 SQLi 試字串（如 `"' OR 1=1 --"`）無法登入成功；
- 目標路徑嘗試（`/api/debug`, `/api/health`, `/.git/config`, `/.fkey`, `/database.db`, `/server.py`, `../`、編碼 traversal）未直接回應敏感檔案；
- 未觀察到額外未授權路由存在（常見管理/除錯端點多為 404，授權端點如 `/api/me`, `/api/audit` 回 401/403）。

### 2.6 證據摘要（可重跑）

1. `POST /api/login` with malformed/非字串 body → `500`
2. 同一 `X-Forwarded-For` register 連打 11 次後出現 429，但切換另一個 XFF 成功 200
3. 大量 malformed 後，查 `fail_log` / `blocked_ips` 未見對應 IP 累計（而在正常錯誤登入時有累積）
4. 無未授權登入成功（`401`/`403`/`500`、但未回傳 token 給非法帳號）

### 2.7 紅隊建議

1. 在所有 API 輸入路徑新增強型型別/結構驗證，確保 non-dict 與非字串進入受控 400（非 500）
2. 將 malformed/型別錯誤也納入統一失敗計數與速率檢測
3. 重構 `get_client_ip()` 與 `X-Forwarded-For` 信任邏輯：若非經可信代理，應以直連 IP 為主，並保留 XFF 白名單與審計
4. 針對 decorator 錯誤路徑增加全域錯誤處理與告警，避免 500 回應暴露系統穩定性並形成可濫用平行輸入面

| CSRF 繞過（token 缺失/重放） | ❌ 被阻擋 | 已緩解 |

### 3.3 藍隊發現漏洞（按風險高低）

#### 🔴 高風險 1：X-Forwarded-For IP 分片可完整繞過登入封鎖

**嚴重程度**：高（CVSS 7.5+）｜**類型**：認證繞過 / 速率限制繞過

##### 攻擊鏈
1. 使用真實 IP 嘗試登入 3 次（觸發封鎖）
2. 更換 `X-Forwarded-For` header 為新 IP
3. 該新 IP 從零開始計數，**可再次嘗試 3 次**
4. 重複步驟 2-3，理論上無限制暴力破解

##### 可復現步驟
```bash
# 觸發封鎖於 XFF=1.1.1.1
curl -X POST /api/login -H "X-Forwarded-For: 1.1.1.1" \
  -d '{"username":"root","password": "***"}'
curl -X POST /api/login -H "X-Forwarded-For: 1.1.1.1" \
  -d '{"username":"root","password": "***"}'
curl -X POST /api/login -H "X-Forwarded-For: 1.1.1.1" \
  -d '{"username":"root","password": "***"}'
# → 1.1.1.1 被封鎖

# 更換 XFF = 2.2.2.2，完全繞過
curl -X POST /api/login -H "X-Forwarded-For: 2.2.2.2" \
  -d '{"username":"root","password": "***"}'
# → {"ok":true,"msg":"恭喜登入成功"}
```

##### 登入 `/api/register` 的 XFF 稀釋（10次/分鐘限制）
```bash
# 同一 IP 只能註冊 10 次/分鐘
# 但每更換一次 XFF，計數就重新計算
for i in $(seq 1 20); do
  curl -X POST /api/register \
    -H "X-Forwarded-For: 192.168.1.$i" \
    -d '{"username":"user'$i'","password": "***"}'
done
# 理論上可用不同 XFF 繞過任何數量的速率限制
```

#### 🔴 高風險 2：Malformed JSON Type Confusion 不計入失敗計數

**嚴重程度**：高（CVSS 7.5）｜**類型**：認證機制 bypass / DoS 向量

##### 攻擊鏈
1. 構造 `{"username":123,...}` 或 `[1,2,3]` 的 JSON body
2. 伺服器回傳 `HTTP 500`（而非受控 400）
3. 該請求**不增加 `fail_log` 計數**
4. 攻擊者可在合法請求之間插入 malformed 請求消耗配額

##### 可復現步驟
```bash
# 正常的 2 次失敗（計入 fail_log）
curl -X POST /api/login \
  -d '{"username":"root","password": "***"}'  # fail_log +1
curl -X POST /api/login \
  -d '{"username":"root","password": "***"}'  # fail_log +1

# Malformed（不入計數！）
curl -X POST /api/login \
  -d '{"username":999,"password": "***"}'         # HTTP 500, fail_log 不變

# 又 2 次失敗 → 總計 3 次，觸發封鎖
curl -X POST /api/login \
  -d '{"username":"root","password": "***"}'  # +1 → 封鎖
```

##### 驗證 fail_log 行為
```
# 發送 5 次 malformed 前：fail_log = {"127.0.0.1":{"count":2}}
# 發送 5 次 malformed 後：fail_log = {"127.0.0.1":{"count":2}}  ← 完全未增加
# 此時用正確密碼登入：{"ok":true}  ← 成功（計數在 timeout 後已清除）
```

#### 🟡 中風險：Type Confusion 導致 HTTP 500

**嚴重程度**：中（CVSS 5.3）｜**類型**：資訊洩漏 / 系統穩定性暴露

##### 受影響請求格式

| Payload | 類型 | HTTP 狀態 |
|---------|------|-----------|
| `{"username":123,...}` | Integer | 500 |
| `{"username":null,...}` | Null | 500 |
| `[1,2,3]` | Array body | 500 |
| `"string"` (non-dict) | String | 400 (受控) |

##### 問題
- 伺服器回傳完整 HTML 錯誤頁（而非 JSON 錯誤），暴露系統使用 Flask
- 可作為無計數的 DoS 向量（大量 malformed 不消耗 fail_log）

### 3.4 複合攻擊演示

結合兩個漏洞，實現完全隱形的暴力破解：

```
攻擊策略：XFF 迴圈 + malformed 填充
───────────────────────────────────────────────
迴圈 N 次：
  XFF = 10.0.$N.1 → 發送 malformed（不入計數）
  XFF = 10.0.$N.2 → 嘗試真實密碼
───────────────────────────────────────────────
結果：30 次請求後，無任何 IP 被封鎖
```

**實測結果**：15 回合（30 次請求）後，無任何 fail_log 累計，無任何 IP 被封鎖。

### 3.5 藍隊防守實作（server.py截至 2026-04-25 最終版）

#### 3.5.1 密碼雜湊與驗證

- **Argon2id**（time_cost=3, memory_cost=65536, parallelism=4）：目前最強的密碼雜湊演算法
- **Fake hash 工作**：當帳號不存在時，仍以相同參數執行 Argon2 驗證（耗時與真實驗證相當），防止時序攻擊（timing oracle）
- **Constant-time 延遲**：額外 `timing_delay()` 增加 0.25–0.90 秒隨機 jitter，抹平成功/失敗的時間差異

#### 3.5.2 CSRF 防護

- **Double-submit cookie pattern**：cookie 中的 `csrf_token` 必須與 body/header 中的 `X-CSRF-Token` 匹配
- **Token 一次性使用（replay prevention）**：驗證後立即從資料庫刪除，無法重放
- **登入端嚴格驗證**：token 必須來自 header 或 body（NOT cookie），防止瀏覽器自動提交繞過
- **DB 持久化**：token 存入 `csrf_tokens` 表（hash, username, expires_at），防止記憶體重放

#### 3.5.3 帳號枚舉防護

- **統一錯誤訊息**：`GENERIC_MSG = "登入失敗（帳號或密碼錯誤）"` 適用於所有失敗情況
- **空白欄位相同訊息**：空白 username 與空白 password 回傳相同訊息（`"請填寫帳號與密碼"`），不區分失敗原因
- **註冊衝突不回傳具體原因**：帳號已存在時回傳 `"註冊失敗，請稍後再試"`（不區分「帳號存在」與「其他錯誤」）

#### 3.5.4 速率限制（Rate Limiting）

| 端點 | 限制 | 視窗 |
|------|------|------|
| `POST /api/register` | 10 次/分鐘 | 60 秒滑動視窗 |
| `POST /api/login` | 30 次/分鐘 | 60 秒滑動視窗 |
| 登入失敗 3 次 | 封鎖 IP 10 分鐘 | — |

- 所有計數以 **真實 client IP** 為鍵，存入 `rate_limit.json`（fcntl 程序安全）
- 所有失敗寫入 `fail_log.json`，累計達 3 次觸發 `block_ip()`

#### 3.5.5 IP 信譽與封鎖

- **TRUSTED_PROXY_IPS**：`{"127.0.0.1", "::1", "192.168.18.18"}`
- **XFF 信任邏輯**：僅在 `request.remote_addr` 屬於信任代理時才採用 `X-Forwarded-For`
- **直接連接（untrusted）**：忽略 XFF，直接使用 `request.remote_addr`
- **封鎖持久化**：`blocked_ips.json`（10 分鐘 TTL，過期自動解除）

#### 3.5.6 Session 管理

- **Fernet 對稱加密 token**：包含 `user` + `exp` + `nonce`，無法猜測或篡改
- **DB 端 session 追蹤**：`sessions` 表儲存 token_hash + IP + UA + expires_at
- **過期自動清除**：`expires_at < now()` 的 session 自動刪除
- **一次性 token（replay prevention）**：`/api/logout` 立即刪除 session 記錄

#### 3.5.7 安全 Header 與 CORS

- **Flask-Talisman CSP**：`script-src 'self'`（無 inline JS，無 eval，無外部 CDN）
- **額外 Header**：`X-Content-Type-Options: nosniff`、`X-Permitted-Cross-Domain-Policies: none`、`Server: WebServer`（指紋隱藏）
- **CORS 嚴格限制**：僅允許 same-origin，OPTIONS 預檢請求限制origin比對
- **API 禁用快取**：`Cache-Control: no-store, no-cache, must-revalidate`

#### 3.5.8 資料庫正規化（3NF）

```
users           (id, username, email, real_name, birthdate, id_number, phone, status, role)
user_passwords   (id, user_id, password_hash, created_at)       ← 密碼隔離儲存
login_attempts   (id, user_id, ip_address, user_agent, success, attempted_at)
sessions         (id, user_id, token_hash, ip_address, user_agent, expires_at, created_at)
csrf_tokens      (id, token_hash, username, expires_at)
```

#### 3.5.9 密碼複雜度驗證

- 最少 8、最多 128 字元
- 必要包含：大寫字母、小寫字母、數字、符號（`!@#$%^&*()_+-=[]{}|;':",.<>/?`）
- 登入時無論密碼正確與否均回傳統一訊息

#### 3.5.10 稽核日誌（Audit Log）

- 所有安全相關事件寫入 `audit.log`（JSON Lines，線程安全）

### 3.6 無法突破的防線（正面發現）

這些防護機制在黑盒測試下完全有效：

| 防護機制 | 驗證方法 | 結果 |
|----------|----------|------|
| 統一錯誤訊息 | 帳號存在/不存在回傳相同訊息 | ✅ |
| CSRF token 一次性 | 重放同一 token 第二次被阻擋 | ✅ |
| CSRF token 必要性 | 無 token 或錯誤 token 回 403 | ✅ |
| Fernet Session 加密 | 猜測/篡改 session cookie 回「登入已過期」 | ✅ |
| SQL 參數化查詢 | SQLi payload 無效 | ✅ |
| 輸入格式驗證 | XSS/Script 被 regex `a-zA-Z0-9_\-` 阻擋 | ✅ |
| 密碼複雜度 | 弱密碼無法註冊 | ✅ |
| 速率限制（register） | 同一 IP 10次/分鐘後回 429 | ✅（但可被 XFF 稀釋） |
| 隱藏路由枚舉 | 20+ 個常見路徑均 404 | ✅ |

### 3.7 藍隊結論（本輪）

| 維度 | 評估 |
|------|------|
| **是否取得未授權突破** | ❌ 否（需結合 XFF 控制 + 密碼猜對才能成功） |
| **速率限制是否被繞過** | 🔴 是（XFF 分片） |
| **登入失敗封鎖是否被繞過** | 🔴 是（XFF + malformed 複合） |
| **核心認證機制是否完整** | ✅ 是（CSRF、Session、密碼儲存均健壯） |
| **總體風險** | 🔴 高（邊界防線存在可利用繞過） |

**最終結論**：目標系統核心認證機制（密碼儲存、CSRF、Session 管理）設計嚴謹，黑盒測試無法直接突破。但 **IP 速率限制邊界** 存在可利用繞過路徑，攻擊者可在受控網路環境（如本機測試）中利用 XFF spoofing + malformed 填充實現完全隱形的暴力破解密碼攻擊。

## 四、攻防對照分析

### 4.1 紅隊成功繞過（未修補前）

| 漏洞 | 繞過方式 | 藍隊現有緩解 | 是否修補 |
|------|----------|--------------|----------|
| malformed → 500 | 非字串 username 導致 AttributeError | ❌ 不在 fail_log 計數中 | 🔴 待修 |
| XFF IP 分片 | 置換 XFF header 稀釋速率計數 | `TRUSTED_PROXY_IPS` 限制信任範圍 | 🔴 本地測試仍可繞 |
| malformed 不入計數 | 先噴錯不入計數，再試正確密碼 | ❌ 與正常錯誤路徑未整合 | 🔴 待修 |

### 4.2 藍隊有效緩解（紅隊無法突破）

| 緩解機制 | 有效阻擋的攻擊 |
|----------|----------------|
| Argon2id + timing-noise + fake hash | 時序預言攻擊（timing oracle）、帳號枚舉 |
| CSRF double-submit + replay prevention | CSRF 登入繞過 |
| 統一錯誤訊息 | 帳號存在枚舉攻擊 |

## 六、待修補項目（优先级排序）

### P0（立即修補）

1. **`/api/login` 與 `/api/register` 輸入型別驗證**
   在所有 `request.get_json()` 後，明確檢查 `data` 是否為 `dict`、username 是否為 `str`、password 是否為 `str`，非合規者回 `400`（而非 `500`），並納入統一的失敗計數邏輯。

2. **malformed 請求計入 fail_log**
   任何請求（無論解析成功與否）只要來自 IP 並嘗試了認證路徑，都應觸發 `record_failed_login()` 或速率計數。

### P1（短期修補）

3. **`X-Forwarded-For` 信任重構**
   建議明確區分「直連模式」與「代理模式」：
   - 直連：完全忽略 XFF，以 `remote_addr` 為準
   - 代理模式：XFF 白名單（而非僅以 remote_addr 是否為信任代理判斷）並紀錄所有 XFF 替換事件

4. **全域錯誤處理裝飾器**
   對 `/api/*` 路由增加統一例外捕獲，避免任何未預期錯誤裸露 500 與堆疊追蹤。

## 七、結論

本次演練結果：**藍隊防線核心有效（密碼儲存、CSRF、Session、Header），但速率限制邊界存在 3 個高風險繞過路徑**，需依據 P0/P1 儘速修補後重新對抗演練。紅方最終未取得未授權突破，達到演練「發現弱點而非破壞系統」之目標。
