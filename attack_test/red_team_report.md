# 紅方攻擊紀錄（web pentest interim report）

專案：`html_learning`  
目標：`http://localhost:5000`（本機）  
時間：2026-04-25（本機本地測試）  
測試者：紅隊（自有授權測試）  
結果：**未取得未授權登入突破（no unauthorized auth bypass）**

---

## 一、攻擊任務與範圍

1. 驗證是否能透過登入 API 越權登入。
2. 壓縮測試速率限制、封鎖機制、CSRF 與 session 行為。
3. 做邊界/異常輸入測試（type confusion、payload 異常、CORS、路徑探測、路由枚舉）。
4. 嘗試敏感檔案與錯誤訊息資源滲透。

---

## 二、總結結論

### 攻擊成效
- 未能成功「**直接登入未授權帳號**」。
- 未能在白名單帳號以外取得有效 session / token。
- 未能透過 SQLi、路徑遍歷、隱藏端點拿到未授權資料。

### 可否作為「攻破」結論
- 依現階段測試結果：**否**（仍屬高風險邊界漏洞存在，但未見完整權限突破）。

---

## 三、紅方發現（高風險）

### 1) `POST /api/login` 未處理非字串 `username`，產生 500（高）

- **現象**：`{"username":123,"password":"x"}`、`{"username":null,...}`、`{"username":["x"],...}`、`[1,2,3]`（非 dict）會得到 `500`。
- **影響**：
  - 進入例外錯誤頁，未經由正常錯誤路徑處理。
  - 這類請求未納入 `fail_log` 失敗計數（可觀察到 60 次 500 後未出現該來源在 `fail_log`）。
- **可復現性**：高，重複可穩定觸發。
- **攻防觀察值**：即使大量噴這類 malformed，也不會啟動封鎖機制，可作為持續壓力/資源消耗向量。

### 2) 以 `X-Forwarded-For` 造成 IP 來源分片（高）

- `get_client_ip()` 在 `remote_addr` 為信任代理（本機常態為 `127.0.0.1`）時，會採用 `X-Forwarded-For`。
- 在本地測試下可觀察到：某一個 XFF 被封鎖後，改用新 XFF 可繼續嘗試/註冊。
- **影響**：
  - `register` 每 10 次/分鐘節流與 `login` 失敗封鎖，可被多重 XFF 快速稀釋。
  - 本地環境中「速率防護」可被來源偽裝繞過成高並行測試窗口。

### 3) malformed 請求與鎖定機制脫鉤（高）

- 使用大量 malformed payload (`[1,2,3]`) 噴向 `/api/login` 時，會大量回 `500`。
- 之後同來源在同 XFF 下測試有效帳密，仍可成功登入（未被該段 malformed 計數導致鎖定）。
- **結論**：失敗告警與封鎖邏輯與輸入驗證錯誤路徑沒有一致性，提升了「先噴錯再試」攻擊可行性。

---

## 四、紅方測試結果（未成功項目）

- `register` 在正常 JSON + 合法條件下可成功，且受節流保護；  
- `'/api/login'` 的 SQLi 試字串（如 `"' OR 1=1 --"`）無法登入成功；  
- 目標路徑嘗試（`/api/debug`, `/api/health`, `/.git/config`, `/.fkey`, `/database.db`, `/server.py`, `../`、編碼 traversal）未直接回應敏感檔案；  
- 未觀察到額外未授權路由存在（常見管理/除錯端點多為 404，授權端點如 `/api/me`, `/api/audit` 回 401/403）。

---

## 五、證據摘要（可重跑）

1. `POST /api/login` with malformed/非字串 body → `500`  
2. 同一 `X-Forwarded-For` register 連打 11 次後出現 429，但切換另一個 XFF 成功 200  
3. 大量 malformed 後，查 `fail_log` / `blocked_ips` 未見對應 IP 累計（而在正常錯誤登入時有累積）  
4. 無未授權登入成功（`401`/`403`/`500`、但未回傳 token 給非法帳號）

---

## 六、紅方結論

- **攻破等級**：未攻破（無未授權登入成功）
- **紅方風險**：存在可持續利用的防護繞過與穩定 DoS/稽核空白面，需盡快修補。

---

## 七、建議（給藍隊）

1. `require_csrf` 與 `login/register` 請求在所有輸入路徑新增強型型別/結構驗證，確保 non-dict 與非字串會進入受控 400（非 500）。  
2. 將 malformed/型別錯誤也納入統一失敗計數與速率檢測。  
3. 重構 `get_client_ip` 與 `X-Forwarded-For` 信任邏輯：若前端未明確經由可信代理，應以直連 IP 為主，並保留 XFF 白名單與審計。  
4. 針對 decorator 錯誤路徑增加全域錯誤處理與告警，避免 500 回應暴露系統穩定性並形成可濫用平行輸入面。
